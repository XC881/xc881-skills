# Research Basis

This skill uses requirements engineering and software-quality planning principles.

## Requirements engineering principles

- Distinguish explicit, inferred, and assumed requirements.
- Write atomic, verifiable requirement statements.
- Separate functional requirements from non-functional requirements.
- Maintain traceability from product goal to requirement to acceptance criteria.
- Make assumptions and unknowns explicit.
- Use acceptance criteria to support verification and validation.
- Analyze requirement dependencies and consequences before implementation.

## Quality categories

Non-functional requirement categories are aligned with common software quality models:

- functional suitability
- performance efficiency
- compatibility
- usability and accessibility
- reliability
- security and privacy
- maintainability
- portability
- observability and operability

## Why this is separate from coding

Requirement analysis is problem-space work.
Coding is solution-space execution.

Separating them prevents premature implementation and gives `$xc881-coding-skills` a clearer source of truth.

## English Spec First

Requirement analysis often starts from rough or mixed-language user intent. Normalizing that intent into a compact English working specification reduces ambiguity before deeper analysis, while preserving the user's original named entities, technologies, paths, and constraints.
