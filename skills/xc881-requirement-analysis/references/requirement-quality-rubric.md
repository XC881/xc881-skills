# Requirement Quality Rubric

Good requirements are:

- necessary
- feasible
- unambiguous
- consistent
- bounded
- testable
- traceable to user or business value
- explicit about assumptions and constraints
- separated from implementation details unless the implementation choice is a real constraint

Bad requirement signals:

- vague verbs such as improve, optimize, support, handle, make better without observable criteria
- mixed product goals and implementation details
- hidden roles or permissions
- no MVP or in-scope boundary
- no out-of-scope boundary
- no acceptance criteria
- no risk list
- no derived requirement analysis
- unranked unknowns
- non-functional requirements that cannot be verified
- architecture chosen before the problem is understood
