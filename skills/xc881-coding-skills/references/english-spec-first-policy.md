# English Spec First Policy

Use when a request is rough, multilingual, ambiguous, emotional, scattered, or needs precise execution.

## Output shape

```text
Standard English Spec:
Goal:
Inputs:
Constraints:
Output:
Process Notes:
```

## Rules

- Keep the spec compact.
- Preserve names, paths, APIs, technologies, metrics, and deadlines exactly.
- Convert vague intent into operational language only when the intent is clear.
- Separate hard requirements from inferred preferences.
- Do not invent requirements.
- If missing information blocks correct execution, ask one concise question.
- If missing information does not block execution, proceed with a labeled assumption.
- Use the spec as the working contract.
- Answer in the user's language unless they request English.
